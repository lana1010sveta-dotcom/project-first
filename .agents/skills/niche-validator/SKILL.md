# Niche Validator — проверка ниши под ключ

Скилл прогоняет бизнес-идею через **6 критериев валидации** силами параллельных субагентов с обязательным факт-чеком. Результат — `REPORT.md` с вердиктом и планом входа.

## Хаб-папка (где живут все проверки)

```bash
HUB="$(pwd)/niche-reports"
```

Все отчёты — в `./niche-reports/reports/<slug>/` в той папке, откуда запущен скилл. Реестр проверок — `./niche-reports/INDEX.md`. Папка создаётся автоматически при старте (см. ниже).

## Зависимости

Проверь на старте:

```bash
# 1. Tavily API key — ОПЦИОНАЛЬНО. Без него скилл работает на встроенном WebSearch.
source ~/.claude/.env 2>/dev/null
if [ -z "$TAVILY_API_KEY" ] || [ "$TAVILY_API_KEY" = "ВСТАВЬ_СЮДА_СВОЙ_КЛЮЧ" ]; then
  echo "⚠️  TAVILY_API_KEY не найден — работаю на встроенном WebSearch."
  echo "    Качество поиска ниже. Бесплатный ключ: https://app.tavily.com → ~/.claude/.env"
  export TAVILY_MODE="off"
else
  export TAVILY_MODE="on"
fi

# 2. Хаб-папка — создаётся автоматически в текущей директории
HUB="$(pwd)/niche-reports"
mkdir -p "$HUB/reports"
```

Скилл стартует в любом случае: с Tavily (точнее) или без него (на WebSearch). Значение `$TAVILY_MODE` передаётся research-агентам — оно определяет, каким поисковиком они пользуются (см. `agents/_contract.md`).

## Даты (используются всеми агентами)

```bash
CURRENT_DATE=$(date +%Y-%m-%d)
CURRENT_YEAR=$(date +%Y)
LAST_YEAR=$((CURRENT_YEAR - 1))
YEAR_BEFORE=$((CURRENT_YEAR - 2))
YEAR_BEFORE_LAST=$((CURRENT_YEAR - 3))
```

Эти значения подставляются в начало промпта каждого Agent вызова отдельным блоком:

```
## КОНТЕКСТ ИСПОЛНЕНИЯ
- Текущая дата: {CURRENT_DATE}
- Текущий год: {CURRENT_YEAR}
- Последний полный год: {LAST_YEAR}
- Год до прошлого: {YEAR_BEFORE}
- Три года назад: {YEAR_BEFORE_LAST}
- Режим поиска (TAVILY_MODE): {on | off} — при `off` основной поисковик WebSearch, Tavily не использовать
```

## ШАГ 0. Recovery check

Если пользователь запускает `/niche-validator` без аргументов — спрашиваешь название идеи и проверяешь существующие папки в `$HUB/reports/`:

```bash
ls $HUB/reports/ 2>/dev/null
```

Если есть незавершённые проверки (наличие `progress.json` без `REPORT.md`):

Через `AskUserQuestion`:
- "Найдена незавершённая проверка [slug]. Что делаем?"
  - Продолжить с этого места (читает progress.json, диспетчеризует только недостающее)
  - Начать новую проверку
  - Только пересинтезировать REPORT (если все 7 critria файлов есть)

## ШАГ 1. Интервью

Открываешь сообщение:
> "Проверяю нишу против 6 критериев. Сначала 8 вопросов — займёт 2-3 минуты."

### Q1 (открытый вопрос — обычное сообщение, не AskUserQuestion)
"Опиши продукт одним предложением: что и для кого."

### Q2 (AskUserQuestion)
**Кто платит?**
- B2C — частники, физлица
- B2B — компании
- B2G — госструктуры, бюджет
- Смешанная — два и более сегмента

### Q3 (открытый)
"Какую главную боль клиента решает продукт? Что они делают сейчас без него — какой workaround? Если не знаешь — пиши 'не знаю'."

### Q4 (открытый)
"Назови 2-5 похожих решений (РФ или мир), которые знаешь. Если никаких не знаешь — пиши 'не знаю'."

### Q5 (AskUserQuestion)
**Гео первого запуска?**
- Россия
- СНГ (РФ + соседи)
- Восточная Европа / Балканы
- Глобально (включая США/Европа)

### Q6 (AskUserQuestion)
**Стадия?**
- Только идея
- MVP / прототип
- Первые клиенты (1-10)
- Стабильная выручка (10+)

### Q7 (AskUserQuestion)
**Ресурсы для запуска?**
- Только я, без бюджета
- Я + до 300к ₽
- Я + команда 2-3 чел
- Я + команда + бюджет

### Q8 (открытый)
"Какие активы для быстрого запуска уже есть? (аудитория / экспертиза / нетворк / продажный канал / список ЛПР). Если ничего — пиши 'нет'."

### Сборка brief.md

```bash
# Слаг — kebab-case из Q1
SLUG="${CURRENT_DATE}-$(echo "$Q1" | head -c 50 | tr '[:upper:]' '[:lower:]' | sed 's/[^a-zа-яё0-9]/-/g' | sed 's/--*/-/g' | sed 's/^-//;s/-$//')"
SLUG_DIR="$HUB/reports/$SLUG"
mkdir -p "$SLUG_DIR"
```

В `$SLUG_DIR/brief.md` пишешь:

```markdown
# Бриф: [название из Q1]
Дата: {CURRENT_DATE}

## Продукт
{Q1}

## Сегмент
{Q2 — B2C/B2B/B2G/смешанный} + детализация из Q1

## Гео
{Q5}

## Боль клиента / текущий workaround
{Q3 или "не указано" если пользователь сказал "не знаю"}

## Известные конкуренты (seed для discovery-агента)
{Q4 или "не указано"}

## Стадия
{Q6}

## Доступные ресурсы
{Q7}

## Активы для быстрого старта
{Q8 или "нет"}
```

### Подтверждение

Покажи `brief.md` пользователю:
> "Вот бриф. Идём в исследование? Или правки?"

Жди подтверждения. После — переход к ШАГУ 2.

## ШАГ 2. Параллельная диспетчеризация — волна 1

Создай `progress.json`:

```json
{
  "slug": "<slug>",
  "started_at": "<current iso datetime>",
  "current_stage": "research_wave_1",
  "agents": {}
}
```

Затем **в одном сообщении** делаешь 6 параллельных `Agent` вызовов (subagent_type: general-purpose):

1. market-size
2. competitors-discovery
3. growth-trend
4. consolidation
5. launch-cost
6. time-to-revenue

Для каждого Agent в `prompt`:
- Сначала блок "## КОНТЕКСТ ИСПОЛНЕНИЯ" с подставленными датами
- Затем содержимое `agents/_contract.md`
- Затем содержимое соответствующего `agents/<criterion>.md`
- Затем блок "## ВХОДНЫЕ ДАННЫЕ — БРИФ" с полным содержимым `brief.md`
- В конце: "## ЗАДАНИЕ — сохрани результат в `<SLUG_DIR>/<criterion>.md`"

**Важно:** все 6 Agent вызовов в одном `<function_calls>` блоке — Claude Code диспетчерит их параллельно.

После завершения всех — парсишь STATUS-код из конца каждого файла, обновляешь `progress.json`.

## ШАГ 2.5. Волна 2 — competitors-deep-dive

Только если `competitors-discovery.md` имеет STATUS = DONE или DONE_WITH_CONCERNS.

Один `Agent` вызов — competitors-deep-dive. В prompt подаёшь дополнительно содержимое `competitors-discovery.md` (full text, не путь).

Обновляешь `progress.json`.

## ШАГ 3. Факт-чек

Один `Agent` вызов — fact-checker. В prompt подаёшь:
- Контекст исполнения (даты)
- Полное содержимое `agents/fact-checker.md`
- Список путей к 7 файлам critria (агент сам их прочитает через Read)
- Инструкция перезаписать каждый файл и создать `fact-check-log.md`

После завершения обновляешь `progress.json`, `current_stage = "synthesis"`.

## ШАГ 4. Синтез (main thread, не Agent)

Сам читаешь:
- `brief.md`
- 7 перезаписанных critria файлов (только VERIFIED факты)
- `fact-check-log.md`

Применяешь **шкалу вердиктов** (см. ниже) к каждому критерию. Считаешь общий вердикт.

Пишешь `REPORT.md`:

```markdown
# Отчёт по нише: {название из brief.md}
Дата: {CURRENT_DATE}

## ВЕРДИКТ
{✅ ГО / ✅ УСЛОВНОЕ ГО / ⚠️ ГРАНИЦА / ❌ НЕ ИДТИ / ⏸ ПАУЗА}

**Одной фразой:** {почему такой вердикт}

## Сводная таблица 6 критериев
| # | Критерий | Статус | Главная цифра |
|---|----------|--------|---------------|
| 1 | Объём рынка | ✅/⚠️/❌ | X млрд ₽ |
| ... |

## По каждому критерию
### 1. Объём рынка — {статус}
**Главные цифры:** ...
**Обоснование вердикта:** ...
**Источники:** ...

[2..6 аналогично]

## Рекомендуемая бизнес-модель
{Выводы из competitors-deep-dive.md о монетизации}

## Топ-5 сильных сторон конкурентов для заимствования
{Таблица из competitors-deep-dive.md}

## Топ-3 риска
{Из всех файлов}

## Топ-3 возможности
{Из всех файлов}

## Рекомендуемая стратегия входа
- Канал первых клиентов: {из time-to-revenue + brief.md активы}
- Реалистичный бюджет: {из launch-cost}
- Срок до первой выручки: {из time-to-revenue}
- Позиционирование: {из competitors-deep-dive: как отстроиться}
- Pre-launch тактика: {если применимо}

## Что проверить вручную
{UNVERIFIED факты из fact-check-log.md, которые могли бы повлиять}

## Приложения
- brief.md
- market-size.md, competitors-discovery.md, competitors-deep-dive.md
- growth-trend.md, consolidation.md, launch-cost.md, time-to-revenue.md
- fact-check-log.md
```

### Шкала вердиктов

| Критерий | PASS | CONDITIONAL | FAIL |
|----------|------|-------------|------|
| 1 Объём | > 5 млрд ₽ за {LAST_YEAR} + 3 независимых источника | 1-5 млрд ИЛИ ≤2 источников | < 1 млрд ИЛИ нет данных |
| 2 Конкуренты | 5+ игроков + новички с подтверждённым ростом + deep-dive дал TOP-5 | 3-5 игроков ИЛИ мало новичков | < 3 игроков ИЛИ все консолидированы |
| 3 Динамика | 4 из 5 индикаторов растут 2 года | смешанные сигналы | 3+ индикатора падают |
| 4 Консолидация | CR3<60% И HHI<2500 | CR3 60-80% ИЛИ HHI 2500-6000 | CR3>80% ИЛИ HHI>6000 ИЛИ 1 игрок >50% |
| 5 Бюджет | реалистично ≤ 300 тыс ₽ | минимум-сценарий без маркетинга | типовое > 1 млн ₽ |
| 6 Срок | кейсы < 2 мес + каналы совместимы с активами | 2-4 мес для совместимых каналов | > 6 мес |

Если по критерию VERIFIED фактов меньше минимума — статус **UNDERRESEARCHED**, не PASS/FAIL.

### Общий вердикт

| Состав | Вердикт |
|--------|---------|
| 6 × PASS | ✅ ГО |
| 5 PASS + 1 CONDITIONAL | ✅ УСЛОВНОЕ ГО |
| 4 PASS + 2 CONDITIONAL | ⚠️ ГРАНИЦА |
| ≤ 3 PASS ИЛИ любой FAIL | ❌ НЕ ИДТИ |
| Любой UNDERRESEARCHED | ⏸ ПАУЗА |

## ШАГ 5. Обновление INDEX.md

Добавь строку в `$HUB/INDEX.md`:

```markdown
| {CURRENT_DATE} | {название} | {вердикт} | {главная цифра} | [reports/{slug}/REPORT.md]({slug}/REPORT.md) |
```

Если строка с тем же slug уже есть — обнови её, не дублируй.

## ШАГ 6. Финал

1. Открой `REPORT.md` в редакторе пользователя: `open "$SLUG_DIR/REPORT.md"`
2. В чате — резюме 3 строки: вердикт + ключевая цифра + следующий шаг
3. Спроси: "Хочешь обсудить какой-то критерий подробнее или сразу переходим к следующему шагу?"

## Правила обработки ошибок

- **Agent failure** → log в `progress.json`, продолжить с partial results
- **Минимум VERIFIED не набрался у агента** → STATUS = DONE_WITH_CONCERNS, критерий в синтезе = UNDERRESEARCHED
- **3+ агента BLOCKED** → останови pipeline, покажи пользователю, спроси что делать
- **Tavily 401/403** → ключ невалидный: НЕ останавливайся, переключи агентов в WebSearch-режим (`TAVILY_MODE=off`) и продолжи; предупреди пользователя, что качество поиска ниже
- **Никаких retry-циклов** — один промах = мимо, иди дальше

## Простой язык

В чате с пользователем — никаких терминов без пояснения. CR3 — "доля трёх крупнейших игроков", HHI — "индекс концентрации", JTBD — "задача клиента", и т.д. Применяется и к содержимому REPORT.md.
